## 1. 사전 작업 (수정 전 현황 파악 — 먼저 보고할 것)

수정에 들어가기 전, 아래를 읽고 표로 보고한다. **이 보고 없이 코드 수정 시작 금지.**

1. **에이전트 프롬프트 위치 확인**
   - 9개 시스템 프롬프트 md 파일 경로 목록 (예: `agents/*.md` 또는 `prompts/*.md`).
   - 각 파일에서 "대화 UI에 출력되는 발화" 가 생성되는 필드명 식별
     (예: `chat_message`, `narration`, `comment` 등 — trace-to-chat-message 변환기가 무엇을 읽는지 확인).
   - 그 발화 필드를 "어떻게 쓰라" 고 지시하는 프롬프트 구간을 특정.

2. **Judge 루브릭 현황**
   - Judge 프롬프트의 **평가 5축 실제 키/라벨** 과 각 축 배점/가중치.
   - 통과 컷(예: 70점) 이 코드/설정 어디에 박혀 있는지.
   - 점수 산출식(weighted) 과 outlier 판정 로직 위치.

3. **저평가 근거 샘플**
   - `runs/` 또는 평가 로그에서 **weighted 70 미만 run 2~3건** 을 찾아,
     어느 축이 점수를 깎았는지(특히 `timeliness_trust` 류) 와 Judge가 남긴 사유를 발췌.

> 보고 양식:
> | 항목 | 위치/값 | 비고 |
> 형태로. 추측 금지, 못 찾으면 "확인 불가" 로 표기.

## Part A. 대화 UI 발화 고도화

### A-1. 문제 정의
현재 발화가 템플릿화·일반화돼 있음. "~까겠습니다", "뭐 이런 거" 같은 멘트가 **콘텐츠 내용과 무관하게 매번 반복**됨 → AI스럽고 영혼 없음.

### A-2. 적용 원칙 (9개 에이전트 공통)
각 에이전트 md의 발화 지침에 아래를 주입한다.

1. **구체 인용 강제:** 발화에는 *방금 처리한 콘텐츠의 실제 디테일* 을 최소 1개 인용/언급.
   일반론("좋은 주제네요", "검토하겠습니다")만으로 끝내지 말 것.
   - 좋은 예(Trend Scout): "5월 가정의 달인데 '목소리만 들어도 다 아는 사이' 같은 감성 후킹이 먹히겠는데. 통화 품질이랑 엮으면 자연스럽고."
   - 나쁜 예: "트렌드를 분석해 주제를 도출하겠습니다."
2. **반복 금지:** 매 run 동일한 오프닝 문구 사용 금지. 표현을 다양화(variation)하라고 명시.
3. **길이:** 발화당 **2~4문장**. 한 문장 단답도, 문단 수준 장황함도 금지(상한 명시).
4. **역할 톤 차별화:** 아래 A-3 톤 가이드를 각 에이전트 발화 지침에 반영.

### A-3. 에이전트별 톤 가이드
| # | 에이전트 | 발화 톤 | 한 줄 지침 |
|---|---|---|---|
| 1 | Trend Scout | 들뜬 발견자 | 방금 찾은 트렌드를 신나서 공유. 왜 지금 이게 뜨는지 한 마디. |
| 2 | Audience Analyst | 냉정한 분석가 | 타깃을 데이터로 읽음. "이 독자층은 ~에 반응" 식 단정. |
| 3 | Strategy Planner | 큰 그림 디렉터 | 주제·타깃을 콘텐츠 각도로 묶는 결정. 방향을 정해주는 톤. |
| 4 | Writer | 몰입한 창작자 | 자기 초안의 의도를 설명. "여기선 ~를 노렸어요" 식. |
| 5 | Fact-Checker | 깐깐한 의심가 | 의심부터. 어떤 사실/출처를 왜 걸었는지 구체 적시. |
| 6 | Devil's Advocate | 삐딱한 반론자 | 도발적으로 약점 찌름. 단, 근거 있는 반론(인신공격 X). |
| 7 | Editor | 균형 조율자 | 양측 의견 정리 후 결단. "둘 다 맞는데 ~로 간다" 톤. |
| 8 | Format Architect | 구조 설계자 | 콘텐츠를 어떤 포맷/슬라이드로 풀지 설계 의도 설명. |
| 9 | HTML Builder | 손 빠른 구현자 | 무엇을 어떻게 구현했는지 담백하게. 과장 X. |

### A-4. 금지 사항 (스키마 보호)
- **JSON 출력 스키마(키 이름·필수 필드·타입) 변경 금지.** 발화 필드의 "내용 생성 지침"만 수정.
- Fact-Checker / Devil's Advocate 는 **톤만** 손대고, 검증·반론 *로직* 은 약화하지 말 것
  (이들이 무르면 콘텐츠 품질이 떨어져 Part B와 충돌).
- `JSON_FORCE_SUFFIX`(prompt 기반 JSON 강제) 구간은 건드리지 말 것 — grounding+json 충돌 회귀 위험.

## Part B. 70점 미만 저평가 개선

### B-1. 진단 먼저 (분기)
저점수의 원인이 둘 중 무엇인지부터 판정해 보고한다.

- **원인 ①(생성 품질):** 콘텐츠가 실제로 약함. 특히 출처 품질
  (미래 날짜 출처 "나무위키 2026-05", 비공신력 블로그성 출처 "아하" 등) → 신뢰도 축 하락.
- **원인 ②(평가 캘리브레이션):** 콘텐츠는 멀쩡한데 Judge가 앵커 없이 보수적으로 깎음.

> ⚠️ **핵심 리스크:** 점수가 낮다고 Judge를 무조건 후하게 풀면 **점수 인플레** → "3-model 교차검증의 객관성" 이라는 셀링포인트가 무너진다. 박함이 정당하면 **콘텐츠를 고쳐야지 Judge를 고치면 안 된다.**

### B-2. 처방 옵션
| 방안 | 내용 | 장점 | 단점 | 권장 |
|---|---|---|---|---|
| A. 생성측 강화 | Writer/Fact-Checker에 출처 검증·필터링 규칙 추가 (미래 날짜·비공신력 출처 배제, 공신력 소스 우선) | 점수가 정직하게 오름. 셀링포인트 강화 | 작업량 ↑ | **채택** |
| B. Judge 앵커 보강 | 5축 각각에 "이 점수=이런 상태" 루브릭 앵커 추가 (가중치·컷은 그대로) | 평가 일관성 ↑, 근거 없는 박함 제거 | 효과 제한적 | **채택** |
| C. 컷·가중치 조정 | 통과 컷 70→하향, 또는 약점 축 가중치 ↓ | 점수 즉시 상승 | 인플레·조작 인상. 셀링포인트 훼손 | **보류(승인 필요)** |

**적용 지시:** B-2-A + B-2-B 만 적용. C(가중치/컷 변경)는 보고만 하고 **사용자 승인 전 적용 금지.**

### B-2-A. 생성측 출처 규칙 (Writer / Fact-Checker md에 주입)
- 출처 채택 기준: 발행 주체가 식별되는 공신력 소스(언론사, 공공기관·통계, 공식 블로그/문서) 우선.
- 배제: ① 현재일(2026-06-05) **이후 날짜** 가 박힌 출처, ② 익명 Q&A·개인 블로그성 출처.
- Fact-Checker는 위 기준 위반 출처를 발견하면 **발화에 구체적으로 적시**하고 교체/삭제 요구.

### B-2-B. Judge 5축 앵커 (Judge 프롬프트에 주입)
- 사전 작업에서 확인한 **실제 5축 키 기준** 으로, 각 축에 3단 앵커를 명시:
  예) `timeliness_trust`: `9~10`=공신력 출처+최신성 명확 / `5~6`=출처 일부 부실 / `1~3`=미래날짜·비공신력 출처 혼입.
- 가중치·통과 컷·산출식은 **변경 금지** (앵커 텍스트만 추가).

## 4. 회귀 점검 (Regression)

- [ ] JSON 파싱 정상 — 9개 에이전트 출력이 기존 converter에서 깨지지 않음.
- [ ] `JSON_FORCE_SUFFIX` / grounding+json 경로 무변경 (빈 응답 회귀 없음).
- [ ] Judge 점수 산출식·outlier 로직 결과가 동일 입력에서 동일 (앵커 추가가 계산을 안 바꿨는지).
- [ ] 백엔드 테스트 전건 PASS.
- [ ] SSE 트레이스 뷰어 정상 렌더 — 발화 길어져도 UI 깨짐/지연 폭증 없음.
- [ ] 발화 토큰 증가로 인한 run 비용·지연 체감 악화 없음(상한 2~4문장 준수 확인).

## 5. 시각 임팩트 체크리스트 (발화가 UI에 노출되므로)

- [ ] 같은 run 안에서 에이전트별 발화 톤이 **구분되게** 읽힘 (9개가 다 비슷하지 않음).

- [ ] 장황해서 스크롤 피로를 주지 않음(상한 준수).

## 6. 분할 commit 안내

작업 후 **2개 commit으로 분리** (Part A / Part B 회귀 추적 용이).

먼저 상태 확인:
```
git status
git diff --stat
```

**Commit 1 — 발화 고도화 (Part A)**
```
git add <에이전트 9개 md 경로>
git commit -m "feat(agents): 발화 구체 인용·톤 차별화로 대화 UI 자연스럽게 개선"
```

**Commit 2 — 평가 개선 (Part B)**
```
git add <Writer/Fact-Checker md> <Judge 프롬프트>
git commit -m "feat(eval): 출처 필터 규칙 + Judge 5축 앵커 추가 (가중치·컷 불변)"
```

> 가중치/컷(방안 C)을 나중에 승인 후 적용하면 별도 3번째 commit으로.
