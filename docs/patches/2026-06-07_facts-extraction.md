# 명세서: `docs/_facts.md` 추출 (발표/심사 자료용 사실 베이스)

산출물: `docs/_facts.md` (단일 파일)

## 0. 목적

예선 심사 자료(`EVALUATION.md`, `역기획서.md`)와 발표 HTML의 **사실 기반(facts base)**을 한 파일에 모은다.
서술·프레이밍·논문 인용·스토리는 별도 작업자(채팅 Claude)가 담당하므로, **이 작업은 코드에 실제로 구현된 사실만** 추출한다.
**추측·창작·미화 금지.** 코드에 근거가 없으면 `[확인 불가]`로 표기한다.

---

## 1. 작업 범위

- **생성**: `docs/_facts.md` (없으면 생성, 있으면 덮어쓰기 전 사용자 확인 위해 `docs/_facts.md`에 그대로 작성)
- **수정 금지**: 위 1개 파일 외 어떤 파일도 생성·수정·삭제하지 않는다.
- **실행 금지**: 빌드·테스트·서버 기동·배포 등 부수효과 있는 명령을 실행하지 않는다. 코드/설정 파일 정적 읽기만 한다. (단, 토큰 카운트 로직 존재 여부 확인을 위한 grep/read는 허용)

## 2. 추출 항목 (각 항목 = `_facts.md`의 한 섹션)

각 사실에는 **근거 위치**(파일 경로 + 함수명/키/라인)를 괄호로 병기한다. 근거 없으면 `[확인 불가]`.

### F1. 9개 에이전트
표로 정리. 컬럼: 에이전트명(한/영) · 소속 스테이지 · 사용 모델 · 입력 · 출력 · 프롬프트 핵심 역할(1~2줄 요약).
- 모델 배치 정확히: 어떤 에이전트가 Gemini 계열이고 어떤 게 Claude 계열인지 (Writer/편집국장 Claude 교체 반영). 실제 모델 문자열까지(`gemini-2.5-flash` 등).
- 근거: `backend/config/personas.yaml` + 모델 호출부 코드.
- `gameifier` 내부 키와 노출 라벨("인터랙티브 빌더")이 다른 점 명시.

### F2. 파이프라인 구조 & 토론 루프
- 3개 스테이지 구성과 각 스테이지 내 에이전트 순서/흐름(순차 vs 루프).
- 토론 루프 파라미터: 최대 반복 횟수, Devil's Advocate 라운드별 가중치(5/3/1), iter 3 강제 승인 + `known_weaknesses` 처리.
- SSE/파이프라인 실행 진입점(`run_manager._run_pipeline` 등) 핵심 흐름.
- 근거: 파이프라인 오케스트레이션 코드 + 관련 설정.

### F3. Judge Panel
- 3개 심사 모델(Gemini/GPT/Claude) 실제 모델 문자열.
- 5축 평가 항목(`topic_fit`, `content_quality`, `interactivity`, `tone_authenticity`, `timeliness_trust`)과 한국어 라벨 매핑.
- 가중·집계 방식, 동시 실행(concurrent) 여부.
- 근거: judge 관련 코드/설정.

### F4. 운영자 입력 최소화 (기획 폼)
- 운영자가 실제 선택하는 입력 = 토픽만. SEG·angle은 시스템 자동회전(round-robin)인 점.
- `planning_presets.json`의 angle 종류 수(9종, `event_tie` disabled 포함)·audience_segments 수(7종).
- Strategy Planner의 angle 주입 + degrade-safe 폴백 동작.
- 근거: `backend/config/planning_presets.json` + PlanningSelector + Strategy Planner 코드.

### F5. 기술 스택 (전체)
- 백엔드: 언어/프레임워크/주요 라이브러리 + 버전(FastAPI, asyncio, SSE, SQLite 등), LLM SDK(`google-genai` 버전), Anthropic SDK 사용 여부/버전.
- 프론트엔드: Next.js 버전, shadcn/ui, Recharts, React Query, Tailwind(다크모드).
- 인프라: Railway(백엔드), Vercel(프론트), Railway Volume `/data`.
- 형상: Git, LF EOL(`.gitattributes`).
- 근거: `requirements.txt`/`pyproject`, `package.json`, 설정 파일.

### F6. 비용 산정용 데이터
- **우선순위 1 (실측)**: 토큰 카운트 로직이 코드에 존재하는지 확인. 존재하면 1 run당 측정 가능한 토큰/호출 데이터 추출.
- **우선순위 2 (추정용)**: 실측 불가 시(UI 0 버그 등), 1 run당 **LLM 호출 횟수**를 구조적으로 카운트: (9에이전트 × 토론 반복) + (Judge 3모델). 모델별 호출 분포(Gemini n회 / Claude n회 / GPT n회)를 표로.
- `cost_is_estimated` 플래그가 코드에 있으면 그 의미(고정 2000/1000 토큰 가정 등) 명시.
- 실측 토큰이 안 나오면 반드시 `[추정 — 실측 불가, 호출 수 기반]`으로 표기.
- 근거: 토큰 카운트/비용 계산 코드, judge 비용 처리부.

### F7. 어드민 옵션 목록
- 현재 어드민에 구현된(또는 UI에 노출된) 설정 옵션 전부 나열.
- **활성 / block(비활성·disabled) 상태 구분.** block 처리된 옵션도 항목명과 의도(있으면)까지.
- 프롬프트 편집·API 키 입력·발행 이력 등 "운영 핸들" 관련 기능 포함.
- 근거: 어드민 프론트 라우트/컴포넌트 + 백엔드 엔드포인트.
- 주: 이 항목은 UI 폴리싱 중이라 변동 가능 — 추출 시점 기준으로 작성하고 상단에 "추출 시점: YYYY-MM-DD" 명시.

### F8. 파일/디렉토리 구조 (핵심만)
- 백엔드·프론트·config·docs 주요 디렉토리 트리(2~3 depth).

## 4. 추출 규칙

1. **근거 우선**: 모든 수치·구성은 코드/설정 근거 병기. 근거 없으면 `[확인 불가]`.
2. **추측 금지**: "아마", "보통" 같은 일반론으로 빈칸 채우지 않는다.
3. **불일치 정직 기록**: 코드와 기존 문서(progress.md 등)가 다르면 부록에 "코드 기준 X, 문서 기준 Y" 병기.
4. **모델 배치 정확**: F1의 모델 컬럼은 실제 호출 코드 기준 (config와 코드가 다르면 코드 우선, 부록에 불일치 기록).
5. **변동 항목 표시**: F7(어드민)은 "추출 시점" 명시.

## 5. 종료 조건 (Termination)

- `docs/_facts.md` 단일 파일이 생성되고, F1~F8 + 부록 섹션이 모두 존재한다.
- 각 섹션은 채워졌거나, 채울 수 없으면 `[확인 불가]`/`[추정]` 사유가 명기되어 있다.
- 위 1개 파일 외 변경된 파일이 없다.
- 빌드/테스트/서버 기동을 실행하지 않았다.

## 6. 회귀 점검 (Regression Check)

- `git status`로 변경 파일이 **`docs/_facts.md` 단 1개인지** 확인한다. 그 외 파일이 변경/생성되었으면 되돌린다.
- 기존 코드·설정 파일의 내용이 바뀌지 않았는지 확인한다(읽기 전용 보장).
- `.gitattributes`의 LF EOL 정책에 맞게 `_facts.md`도 LF로 저장되었는지 확인한다.
