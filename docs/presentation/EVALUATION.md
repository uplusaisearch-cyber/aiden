# AIDEN — 평가용 기술 요약

> 본 문서는 AI 심사를 포함한 평가자가 시스템을 빠르게 파악하도록 작성한 자족적 요약이다.
> 모든 구성은 실제 구현 기준이며, 미구현·추정 항목은 명시했다.

## 1. 시스템 개요

AIDEN(AI Deliberation Engine for Newsroom)은 단일 생성 프롬프트가 아니라, 9개 에이전트가 3개 뉴스룸으로 분업하고 3개 심사 모델이 교차 채점하는 **콘텐츠 제작 플랫폼**이다.

운영자 입력 = 토픽 카테고리 1개. 앵글(8종 활성)과 독자 세그먼트(7종)는 round-robin으로 자동 배정되며, 수동 지정도 가능하다.

End-to-end 파이프라인:
```
[입력: 토픽 카테고리]
  → Stage 1 토픽 뉴스룸 (정찰 → 분석 → 기획)
  → Stage 2 콘텐츠 뉴스룸 (작가 ↔ 팩트체커 ↔ 반론 ↔ 편집국장, 최대 3라운드 토론)
  → Stage 3 인터랙티브 빌더 (포맷 설계 → HTML 생성)
  → Stage 4 심사단 (Gemini·GPT·Claude 동시 5축 채점)
[출력: 즉시 반영 가능한 HTML + 검증 트레이스 + 점수]
```

## 1.5 시스템 아키텍처 (구현 구조)

콘텐츠 파이프라인(§2)이 "무엇을 만드나"라면, 본 절은 "어떤 컴포넌트가 어디서 어디로 호출되나"다. 단일 LLM 래퍼가 아니라, 프론트·백엔드·실시간 스트림·영속 저장이 분리된 실제 웹 서비스 구조다.

```
[브라우저]
   │
   ▼
[Vercel] Next.js 14 (App Router, shadcn/ui, React Query)
   │  ① POST /api/generate ───────────────▶ 즉시 session_id 반환
   │  ② SSE 구독 /api/stream ◀───────────── 실시간 트레이스 push
   ▼
[Railway] FastAPI 백엔드
   │
   ├─ RunManager : POST 수신 → asyncio.create_task 로 파이프라인을 백그라운드 실행
   │     (요청-응답을 막지 않음. 실행은 비동기로 분리)
   │
   ├─ FullPipeline (sync) ← asyncio.to_thread 로 호출
   │     ├─ Stage 1 TopicNewsroom   (순차)
   │     ├─ Stage 2 ContentNewsroom (최대 3-iter 토론 루프)
   │     ├─ Stage 3 Gameifier       (순차, 실패 시 fallback HTML)
   │     └─ Stage 4 JudgePanel      (asyncio.gather 로 3모델 동시)
   │
   ├─ 9개 에이전트 → LLM Provider SDK 직접 호출
   │     ├─ google-genai  → Gemini (flash / pro / 3.1-pro), Grounding=GoogleSearch
   │     ├─ anthropic     → Claude (sonnet-4-6 / opus-4-7)
   │     └─ openai        → GPT (gpt-5, judge 전용)
   │
   ├─ SSE Broker : 각 에이전트 단계마다 pub/sub 으로 프론트에 push
   │
   └─ SQLite (outputs.db) : 완료된 run 만 영속 저장 (Railway Volume /data 마운트)
```

### 호출 흐름 (요청 1건 기준)

1. 프론트가 `POST /api/generate` 로 토픽을 전달 → 백엔드는 즉시 `session_id` 만 반환(논블로킹).
2. `RunManager` 가 `asyncio.create_task` 로 파이프라인을 백그라운드 기동. 동기 `FullPipeline` 은 `asyncio.to_thread` 로 이벤트 루프를 막지 않게 실행.
3. 각 에이전트 단계가 끝날 때마다 `SSE Broker` 가 트레이스 이벤트를 `/api/stream` 구독자(프론트)로 push → 실시간 진행 표시.
4. 9개 에이전트는 alias→실모델 매핑(`config/agents.yaml`)에 따라 google-genai / anthropic / openai SDK를 직접 호출. Grounding 대상은 Google Search 도구를 붙여 호출.
5. Stage 4 심사단은 `asyncio.gather` 로 3개 모델을 동시 호출하고, timeout·부분 실패를 `degraded` 상태로 처리.
6. 파이프라인 완료 시 `final_output.html` 저장 + `outputs.db` upsert + `pipeline_complete` 이벤트 발행. 타임아웃 가드 1200초.

### 구현상 난이도 포인트

- **비동기 ↔ 동기 경계:** 백그라운드 태스크 + `to_thread` + `gather` 가 혼재. 동기 파이프라인을 이벤트 루프와 분리해 SSE 응답성을 유지.
- **실시간 스트리밍:** SSE 브로커를 자체 구현(pub/sub). 에이전트 단계별 이벤트를 세션 단위로 라우팅.
- **다중 SDK 통합:** 3개 벤더 SDK의 호출·인증·JSON 파싱·폴백을 한 파이프라인에 통합. Gemini 폴백 체인(flash→flash-lite)과 Grounding+JSON 충돌 우회 포함.
- **안전 치환:** HTML 빌더가 화이트리스트 기반으로 placeholder만 치환해 임의 마크업 주입 차단.
- **degrade-safe:** 각 에이전트·심사 모델 실패가 전체 run을 깨지 않도록 부분 결과·강제 승인·fallback HTML로 격리.

### 2.1 9개 에이전트 구성

| # | 에이전트 | Stage | 모델 | 그라운딩 | 역할 |
|---|---|---|---|---|---|
| 1 | 트렌드 정찰 | 토픽 뉴스룸 | gemini-2.5-flash | ON | 카테고리 핫이슈 실시간 수집, 발행 토픽 회피 |
| 2 | 독자 분석 | 토픽 뉴스룸 | gemini-2.5-flash | OFF | 토픽별 페르소나 적합도 평가 |
| 3 | 편집 기획 | 토픽 뉴스룸 | gemini-3.1-pro-preview | OFF | 최종 토픽·앵글·타겟 확정 |
| 4 | 작가 | 콘텐츠 뉴스룸 | claude-sonnet-4-6 | OFF | 초안 작성·라운드별 재작성 |
| 5 | 팩트체커 | 콘텐츠 뉴스룸 | gemini-2.5-flash | ON | 인용·수치·도메인 검증, 인라인 주석 |
| 6 | 악마의 대변인 | 콘텐츠 뉴스룸 | gemini-2.5-flash | OFF | 라운드별 비판(5/3/1건) |
| 7 | 편집국장 | 콘텐츠 뉴스룸 | claude-opus-4-7 | OFF | 비판 수용/기각, 통과·재작성·직접편집 결정 |
| 8 | 포맷 설계 | 인터랙티브 빌더 | gemini-2.5-pro | OFF | A/B/C 타입 + 위젯 후보 선택 |
| 9 | HTML 빌더 | 인터랙티브 빌더 | gemini-2.5-pro | OFF | 마크업 생성 + 화이트리스트 치환 |

모델은 노드 중요도·호출 빈도에 따라 의도적으로 배치했다. 최종 품질을 좌우하는 편집국장에 최상위 모델(Opus), 반복 호출이 많은 정찰·검증·반론에 경량 모델(Flash).

### 2.2 토론 루프 (검증의 핵심)

콘텐츠 뉴스룸은 단방향 생성이 아니라 최대 3라운드 토론이다.

- 작가 초안 → 팩트체커 검증(인라인 주석) → 악마의 대변인 비판 → 편집국장 결정
- 편집국장이 `approved`면 즉시 종료, 아니면 다음 라운드
- 악마의 대변인 비판 건수는 라운드별 5 → 3 → 1로 수렴
- **3라운드 강제 종료 + 투명성:** 3라운드에서도 미승인이면 오케스트레이터가 강제 통과시키되, 잔여 비판과 팩트체커 저신뢰 사유를 `known_weaknesses`로 명시. 무한 루프 방지 + 약점 은폐 방지.
- **자가검증:** 편집국장이 "직접 수정함"이라 표시했는데 실제 텍스트 변경이 없으면(유사도 ≥ 0.99) 경고를 `known_weaknesses`에 추가.

이 구조의 목적은 정답률이 아니라 **검증 과정의 가시화**다. 전 과정이 SSE 실시간 트레이스로 노출된다.

### 2.3 자동화 범위

입력(토픽 1개)부터 출력(반영 가능 HTML)까지 사람 개입이 없다. 앵글·세그먼트 자동 배정, 그라운딩 기반 토픽 선정, 토론 합의, 포맷 결정, HTML 생성, 안전 치환이 모두 파이프라인 내부에서 처리된다.

## 3. [10점] 콘텐츠 유용성

- **실데이터 기반.** 트렌드 정찰과 팩트체커가 Google Search 그라운딩으로 실시간 정보를 수집·검증한다. 학습 데이터 회상이 아니라 현재 시점 데이터다.
- **인라인 검증.** 팩트체커가 본문의 인용·수치·도메인에 주석을 달고 신뢰도 점수를 매긴다. 저신뢰 항목은 편집 단계와 `known_weaknesses`로 이어진다.
- **발행 회피.** 이미 발행한 토픽을 정찰 단계에서 회피해 중복을 줄인다(백엔드 동작).

---

## 4. [10점] 콘텐츠 완성도

- **즉시 반영 가능한 HTML.** 산출물은 텍스트가 아니라 A/B/C 콘텐츠 타입에 맞춘 HTML이다. 샘플 콘텐츠의 구조·톤을 차용했다.
- **안전 치환.** HTML 빌더가 화이트리스트 기반으로 placeholder를 치환해 임의 코드 주입을 차단한다.
- **외부 요구사항 수용.** CTA 링크·이미지 등 운영 요구사항을 받아들이는 치환 구조가 있어, 요구사항이 생기면 코드 수정 없이 반영 가능하다.

산출 HTML은 PlusTab 샘플의 HTML 골격·클래스 구조를 차용하고, CTA·이미지 등 운영값은 화이트리스트 치환 슬롯으로 주입한다. 임의 마크업을 새로 만들지 않으므로 운영 디자인과의 정합을 유지한다. 서비스 디자인 가이드의 완전 적용·재검증은 운영 이관 단계 과제다(§9.1).

---

## 5. [10점] 독창성·완결성

1. **멀티모델 교차검증.** 심사단이 단일 벤더가 아닌 Gemini·GPT·Claude 3사 동시 채점. 점수 분산 자체가 교차검증 신호다.
2. **검증 가시화.** 작성·검증·반론·편집을 분리하고 토론 과정을 실시간 노출. "결과만"이 아니라 "의심하고 고치는 과정"을 보여준다.
3. **합의 기반 종료.** 강제 통과 시에도 약점을 투명하게 남기는 설계.

설계 동기: 멀티에이전트 토론이 정답 추론 과제에서 비용 대비 단일 에이전트보다 우월하지 않다는 연구가 있다. AIDEN은 그 토론을 정답 추론이 아니라 콘텐츠 검증 가시화에 적용해, 다른 도메인에서의 유효성을 실험한 결과물이다.

## 6. 심사단 (Judge Panel)

| Judge | 모델 | 호출 |
|---|---|---|
| Gemini | gemini-2.5-pro | concurrent |
| GPT | gpt-5 | concurrent |
| Claude | claude-opus-4-7 | concurrent |

5축 평가(가중치 합 100): 타깃 적합성(topic_fit, 20) · 콘텐츠 품질(content_quality, 25) · 인터랙티브(interactivity, 15) · 톤 진정성(tone_authenticity, 20) · 출처 신뢰(timeliness_trust, 20). 세 모델 점수의 평균·표준편차를 산출하고, 표준편차가 큰 축은 outlier로 표시한다.

## 7. 비용 (현재 모델·테스트 기준 추정)

실측이 아닌 호출 수 기반 추정이다. 콘텐츠 1건당 LLM 호출은 토론 라운드에 따라 12~20회.

| 라운드 | 총 호출 수 |
|---|---|
| 1 | 12 |
| 2 | 16 |
| 3 | 20 |

비용 대부분은 편집국장(Opus, 라운드당 1회)과 심사단(Opus·GPT)에서 발생한다. 1건당 약 500원 내외로 추정된다(라운드 수에 따라 변동). 운영 전환 시 편집국장·심사단을 경량 모델로 내리면 건당 비용을 대폭 절감할 수 있다.

(주: 현재 9에이전트 토큰 실측은 비활성 상태로, UI 표시 수치 대신 위 호출 수 기반 추정을 사용한다.)

## 8. 기술 스택

- **백엔드:** Python(FastAPI), asyncio, SSE 자체 구현, SQLite. SDK: google-genai·openai·anthropic.
- **프론트엔드:** Next.js 14, shadcn/ui, Recharts, React Query, Tailwind(다크모드).
- **인프라:** Railway(백엔드) + Vercel(프론트), 영속 볼륨 마운트 예약.
- **형상:** Git(LF EOL 강제).

## 9. 한계 및 로드맵 (v2)

현재 버전은 데모-운영 사이의 다리를 목표로 했다. 운영 이관 시 채워야 할 항목을 영역별로 구체화한다. 한계를 명시하는 이유는, 추가 작업 범위가 명확해야 실제 이관 가능한 시스템이기 때문이다.

### 9.1 콘텐츠 품질·스타일

| 항목 | 현재 | v2 방향 |
|---|---|---|
| 서비스 CSS 적용 | 샘플 골격·클래스 차용, 운영값은 치환 주입 | 실제 서비스 스타일 적용본 검증·수정, 생성 구조의 운영 적합성 재확인 |
| 톤·브랜드 정합 | 기존 콘텐츠 톤 차용 | 콘텐츠 스쿼드 협의 후 브랜드 색채 강화 여부 결정, 가이드 반영 |
| 랜딩/이벤트 연결 | 미구현 | 산출 HTML의 CTA를 앱 내 랜딩·이벤트 페이지로 연결 |
| 게이미피케이션 샘플 | 일부 위젯 타입 | 자동 주입 샘플 확대 |

### 9.2 참조 정보·검색(RAG)

| 항목 | 현재 | v2 방향 |
|---|---|---|
| 생성 시점 정보 입력 | 슬롯 예약 | generate 시 이벤트·캠페인 정보를 운영자가 직접 주입(컨텍스트 주입) |
| 사전 문서 주입 | 미구현 | 어드민에 참조 문서를 수동 등록 → 에이전트가 참조 |
| 발행 이력 활용 | 백엔드 동작, UI 폐쇄 | 발행 이력 자동 적재 + UI 노출, 중복 회피 고도화 |
| 토픽 유사도 중복 제거 | 미구현 | 임베딩 기반 유사도로 발행 토픽 중복 탐지(dedup window) |
| 문서 검색 RAG | 미구현 | 참조 문서가 늘면 임베딩 검색 기반 RAG로 전환 |
