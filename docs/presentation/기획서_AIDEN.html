AIDEN

AI Deliberation Engine for Newsroom
운영자는 토픽만 고른다. 나머지 제작 공정과 확장은 플랫폼이 맡는다.

한눈에 왜 만들었나 역할 분담 동작 방식 설계 결정 만든 과정 운영 이관 비용 기술 스택
00

한눈에

AIDEN은 콘텐츠 생성기가 아니다. 콘텐츠 제작 플랫폼이다.

운영자가 토픽 카테고리 하나만 누르면 — 트렌드 정찰, 사실 검증, 편집 합의, 그리고 바로 쓸 수 있는 HTML 산출까지 — 뉴스룸의 제작 공정 전체가 자동으로 돈다.

이름의 Deliberation(토론·합의) 이 전부다. 한 모델에게 "글 써줘"가 아니라, 편집국이 토론하고 합의해서 콘텐츠를 만드는 과정을 통째로 자동화했다.

토픽 하나 선택
   │
   ▼
[정찰 → 분석 → 기획]              무엇을 쓸지 정한다
   │
   ▼
[작가 ↔ 팩트체커 ↔ 반론 ↔ 편집국장]   토론하며 다듬는다 (최대 3R)
   │
   ▼
[포맷 설계 → HTML 생성]           바로 쓸 수 있게 만든다
   │
   ▼
[Gemini · GPT · Claude 동시 채점]   세 모델이 교차 검증한다
   │
   ▼
완성 HTML + 검증 과정 전체가 화면에 남는다
01

왜 이렇게 만들었나

LLM에게 글을 시키는 건 이제 기본기다. AI가 글을 쓰고 에이전트끼리 대화하는 데모도 이미 넘친다. 그래서 그게 정말 유효한지 직접 확인하고 싶었고, 두 가지에 걸었다.

하나, 믿을 수 있는 품질

AI 글쓰기의 약점은 늘 같다 — 할루시네이션, 근거 없음. 이걸 결과물 뒤에 숨기는 대신, 작성·검증·반론·편집을 서로 다른 역할로 쪼개서 서로 공격하게 만들었다. 글이 어떻게 깨지고 고쳐지는지가 그대로 보인다.

둘, 진짜 넘길 수 있는 시스템

데모로 끝내고 싶지 않았다. 어드민, 프롬프트 편집, 출력 이력까지 붙였다. 심사 기준의 "즉시 서비스 반영"을 말이 아니라 동작으로 보여주는 게 목표였다.

곁다리로 궁금한 것도 하나 풀어봤다. "여러 AI가 토론하는 게 혼자 하는 것보다 비용 대비 낫지 않다"는 연구가 있다. 다만 그 연구들은 정답 맞히기가 대상이다. 콘텐츠 생산이라는 다른 판에서도 그런가? AIDEN의 토론은 정답을 맞히려는 게 아니라 검증 과정을 드러내려는 거라, 애초에 결이 다르다.
02

내 역할 / AI의 역할

설계·검증·운영은 내가, 구현은 AI가.

코드는 내가 직접 짜지 않았다. 한 줄도. 하지만 전부 내가 설계했다.

내가 한 것

  • 아키텍처 설계
  • 검증 루프 정의
  • 역할별 모델 배치 판단
  • 운영 의사결정
  • GitHub · 서버 · 배포 · 테스트

AI가 한 것

  • 코드 구현 (Claude / Claude Code)
  • 9에이전트 + 3심사단 로직
  • 프론트엔드 · 백엔드 · 파이프라인

무엇을 만들지, 어떤 검증을 거칠지, 어느 노드에 어느 모델을 쓸지 — 이 결정들의 연쇄가 곧 이 플랫폼이다. 그 결정이 기획의 몫이었다.

03

어떻게 돌아가나

3개 뉴스룸 + 선택적 심사단. 운영자 입력은 토픽 하나.

운영자가 하는 일은 딱 하나

토픽 카테고리 1개 선택. (맛집 · AI트렌드 · 안전 · 문화 중 하나, 또는 자유 입력) 끝.

전개 방식(앵글)과 타겟 독자는 플랫폼이 알아서 돌려가며 배정한다. 앵글 8종, 독자 7종이 순환한다. 원하면 직접 고를 수도 있지만, 기본은 "토픽만 주면 알아서"다. 이게 플랫폼의 핵심 사용 경험이다.

① 토픽 뉴스룸 — 무엇을 쓸지 정한다

  • 정찰 : Google 검색으로 카테고리 핫이슈를 실시간 수집. 이미 발행한 토픽은 피한다.
  • 분석 : 토픽이 타겟 독자에게 맞는지 점수를 매긴다.
  • 기획 : 최종 토픽 · 앵글 · 타겟을 확정한다.

② 콘텐츠 뉴스룸 — 토론하며 다듬는다 (최대 3라운드)

여기가 AIDEN의 심장이다.

  1. 작가가 초안을 쓴다.
  2. 팩트체커가 인용·수치·출처를 검증하고 본문에 주석을 단다.
  3. 악마의 대변인이 논리 비약·식상함·근거 부족을 물고 늘어진다. (1R 5건 → 2R 3건 → 3R 1건. 초반엔 세게 흔들고 후반엔 수렴시킨다.)
  4. 편집국장이 비판을 받아들이거나 기각하고, 통과·재작성·직접 수정을 결정한다.
3라운드까지 가도 결론이 안 나면 편집국장이 강제로 통과시킨다. 단, 남은 약점을 known_weaknesses로 숨기지 않고 명시한다. 무한 루프도, 약점 은폐도 막는 장치다.

③ 인터랙티브 빌더 — 바로 쓸 수 있게 만든다

  • 포맷 설계 : A(기본형) · B(복합형) · C(기타) 중 선택 + 인터랙티브 위젯 후보.
  • HTML 생성 : 실제 마크업을 만들고, 화이트리스트로 안전하게 치환한다.

그리고 세 명의 심사위원

완성된 HTML을 Gemini · GPT · Claude 세 모델이 동시에 5개 기준(타깃 적합성 · 콘텐츠 품질 · 인터랙티브 · 톤 진정성 · 출처 신뢰)으로 채점한다. 한 회사에 채점을 맡기지 않는다. 세 점수가 갈리면, 그 차이 자체가 교차검증 신호다.

04

설계에서 신경 쓴 것

모델은 자리에 맞춰 앉혔다

전부 같은 모델을 쓰면 편하지만 안 그랬다. 노드의 중요도와 호출 빈도로 갈랐다.

자리모델왜
편집국장Claude Opus최종 품질을 좌우하는 단 하나의 노드. 여기만큼은 최강 모델.
작가Claude Sonnet발화 품질이 글의 첫인상을 정한다.
편집 기획Gemini 3.1 Pro전략 판단이 핵심.
정찰·검증·반론Gemini Flash호출이 잦다. 경량 모델로 비용을 눌렀다.
심사위원Gemini·GPT·Claude한 벤더에 쏠리지 않게.

처음엔 9개 전부 Gemini였다. 작가와 편집국장만 Claude로 바꿨다. 발화 품질과 편집 판단에서 눈에 띄게 차이가 났기 때문이다. 이 교체 자체가 "어느 자리에 어느 모델이 맞나"를 실험한 결과다.

토론을 왜 넣었나

정답률을 높이려는 게 아니다. 검증 과정을 사용자가 눈으로 보게 하려는 것이다. 작가의 초안이 어떻게 공격받고 고쳐졌는지가 실시간으로 화면에 흐른다. "AI가 결과만 툭 던진다"가 아니라 "AI가 스스로 의심하고 고친다" — 이게 신뢰의 출발점이라고 봤다.

05

만든 과정

가설 → 설계 → 검증/재설계 → 운영화. 막힌 지점은 원인까지 팠다.

  1. 던지기 — 아이디어를 풀어놓고, "예선은 AI도 심사한다"는 조건을 확인한 뒤 평가에 맞는 구조로 다시 잡았다.
  2. 가설 — 멀티에이전트 토론이 콘텐츠 품질에 정말 기여하나?
  3. 설계 — 9에이전트 / 3라운드 토론 / 심사단.
  4. 검증하며 고치기 — 실시간 트레이스가 가끔 비는 버그가 있었는데, 원인이 React 상태 업데이트의 순수성 위반이라는 데까지 추적해 고쳤다. 그라운딩과 JSON 강제 출력이 충돌하는 SDK 제약은 프롬프트 기반 우회로 풀었다. 작가·편집국장 모델 교체도 이 단계의 판단이다.
  5. 운영화 — 어드민, 출력 이력, 프롬프트 편집을 붙였다.

막힌 지점들은 삽질이 아니라 원인을 끝까지 판 기록이다. 이게 production 시스템과 데모를 가르는 차이라고 본다.

06

운영에 넘긴다면

코드를 안 만져도, 정해진 방향대로 디벨롭할 수 있게.

이 플랫폼은 "한 번 만든 데모"가 아니라 계속 키울 수 있는 베이스를 지향했다.

지금 열려 있는 확장 포인트

  • 요구사항 주입 — CTA 링크, 이미지처럼 콘텐츠에 들어가야 할 외부 요구사항을 받는 치환 구조가 이미 있다. 요구가 생기면 코드 없이 끼운다.
  • 참조 정보 주입 — 특정 이벤트·캠페인을 참고해야 할 때, 에이전트가 볼 정보를 넣을 슬롯을 예약했다. 발행 이력 기반 토픽 회피는 백엔드가 이미 동작한다(UI는 v2).
  • 프롬프트 운영 — 9에이전트 + 3심사단의 프롬프트를 어드민에서 직접 고치고, 버전 관리하고, 되돌릴 수 있다. 톤이나 전략을 바꾸려면 코드가 아니라 프롬프트만 손대면 된다.

아직 남은 한계 (솔직하게)

  • CSS — 지금은 샘플로 받은 HTML 구조 그대로 나온다. 실제 서비스 스타일을 입힌 버전은 따로 검증·수정해야 한다.
  • 영속성 — 프롬프트·키·실행 이력이 아직 휘발성이다(재배포 시 초기화). 영속 저장으로의 전환이 v2 과제.
  • 운영 옵션 UI — 설정 화면의 틀과 슬롯은 잡혀 있지만, 서버에 실제로 반영하는 연결은 아직 범위 밖이다.
한계를 적어두는 이유는 단순하다. 무엇을 더 해야 넘길 수 있는지가 명확해야 진짜 이관 가능한 시스템이라서다.
07

비용

현재 모델 구성 · 테스트 기준 추정으로, 콘텐츠 1건에 약 500원 내외다(토론 라운드 수에 따라 변동).

비용 대부분은 편집국장과 심사단에 둔 프리미엄 모델에서 나온다. 의도한 설계다. 품질을 좌우하는 최종 노드에만 최강 모델을 쓰고, 호출이 잦은 정찰·검증·반론은 경량 모델로 눌렀다. 운영 전환 때 이 노드들만 경량화하면 건당 비용은 더 내려간다.

기술 스택

영역구성
백엔드Python(FastAPI), asyncio, SSE 자체 구현, SQLite / SDK: google-genai · openai · anthropic
프론트엔드Next.js 14, shadcn/ui, Recharts, React Query, Tailwind(다크모드)
인프라Railway(백엔드) + Vercel(프론트), 영속 볼륨 마운트 예약
형상Git (LF EOL 강제)

소스 코드 전체는 공용 계정 GitHub에 있고 운영 조직으로 이관 가능하다. API 키는 현재 테스트용 개인 계정에 물려 있어, 이관 시 운영 계정 전환이 필요하다.
